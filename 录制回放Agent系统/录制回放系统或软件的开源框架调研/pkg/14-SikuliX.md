# SikuliX / Oculix

## 项目介绍

SikuliX 是经典的视觉自动化和 GUI 测试工具，核心理念是“自动化屏幕上能看到的东西”。它通过 OpenCV 图像识别在屏幕截图中定位按钮、图标、菜单、文本区域等视觉元素，然后用鼠标和键盘模拟完成操作。

SikuliX 最适合处理没有 DOM、没有标准控件树、无法通过 UIA/Win32/Selenium 选择器稳定识别的界面，例如老旧 CS 客户端、自绘控件、远程桌面、虚拟机、图形化行业软件等。其原始 SikuliX1 仓库在 2026 年转为历史镜像，后续活跃维护方向主要迁移到 Oculix。

## 录制方式

- **主要方式：基于截图模板的视觉脚本。** 用户在 IDE 中截取按钮、菜单、图标或区域图片，然后编写点击、等待、输入、查找等脚本。
- SikuliX IDE 支持用截图嵌入脚本，降低视觉步骤的编写门槛。
- 支持 Python/Jython 风格脚本，也可与 Java、Robot Framework、其他自动化框架组合。
- **不是视频录制。** 它不会分析录屏生成流程，而是依赖截图模板和脚本命令进行回放。
- **不是 AI 语义理解。** 图像匹配能找到“看起来一样”的区域，但不知道业务含义，也不会根据任务目标自主探索 App。
- 参数化可以通过脚本变量、外部文件、循环和框架集成实现，但需要人工设计。

## 核心能力

| 能力 | 结论 |
| --- | --- |
| AI 语义理解 | 不支持 |
| 框架自动截图 | 支持，截图是核心工作方式 |
| 图像识别 | 支持，基于 OpenCV |
| OCR | 可扩展支持，Oculix 方向提到 PaddleOCR 选项 |
| UIA/原生控件信息 | 不作为核心能力，主要看屏幕图像 |
| 人工录制 | 不是真正录制器；主要是截图模板 + 脚本编辑 |
| 生成回放资产 | 支持，视觉脚本和图片资源 |
| 参数化和批量执行 | 支持脚本级变量、循环和外部数据读取，需要人工实现 |
| 一键回放 | 支持 |
| 自动探索 App | 不支持 |
| 适用平台 | Windows、macOS、Linux |
| 开源 | 是，MIT |

## 优势

- 对“控件树不可见”的应用非常有价值，只要屏幕上可见就有机会自动化。
- 适合远程桌面、虚拟机、老旧业务系统、自绘 UI、图形化客户端等场景。
- 视觉脚本直观，截图模板能让非底层开发人员理解定位目标。
- 可作为 OpenRPA、TagUI、Robot Framework、Python 自动化方案的视觉兜底层。
- 跨平台，可覆盖 Windows、macOS、Linux。

## 局限和风险

- 对分辨率、缩放比例、主题颜色、字体、语言、窗口位置和图标变化敏感。
- 图片模板越多，维护成本越高；界面改版后可能需要批量更新截图资源。
- 无法天然理解控件层级、表格结构、业务字段和异常语义。
- 长流程需要大量等待、重试、异常分支和截图校验，否则稳定性不足。
- 不适合单独承担高频批量生产任务，更适合作为视觉补位能力。

## 不符合要求的核心原因

1. **不具备 AI 语义理解和自主探索 App 能力。**
2. **没有完整“人工录制 -> 自动生成参数化脚本”的产品化闭环。**
3. **核心定位依赖截图模板，稳定性弱于 UIA/Win32 控件级定位。**
4. 参数化、日志、批量调度、失败重跑等生产能力需要外部框架补齐。

## 适用结论

SikuliX / Oculix 适合作为桌面自动化中的视觉兜底层，尤其适合老旧客户端、远程桌面、自绘控件和控件树不可访问场景。不建议作为本次目标的主方案。

推荐组合方式是：Robot Framework / Python / OpenRPA 负责流程编排、参数化、日志和批量执行；pywinauto/FlaUI/UIA 负责能识别的原生控件；SikuliX/Oculix 负责最后一层图像识别和视觉点击。
