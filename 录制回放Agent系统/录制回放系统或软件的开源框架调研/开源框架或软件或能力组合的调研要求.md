# 开源框架或软件或能力组合的调研要求

## 1. 调研背景

本次调研关注的是一类面向 Windows 桌面软件的自动化能力：系统能够理解用户给出的业务目标，自动探索桌面软件界面和操作路径，并将探索或示教过程中形成的操作流程沉淀为可复用、可参数化、可稳定回放的脚本或工作流。

这里的重点不是单纯录屏，也不是只记录鼠标坐标，而是希望找到一套更接近“AI 探索 + 流程固化 + 批量回放”的方案。最终脚本应尽量减少人工参与，能够在不同输入参数下重复执行同一类数据获取、查询、导出或处理任务。

## 2. 核心目标

调研对象可以是单一开源框架、开源软件，也可以是多个能力组合形成的方案。优先考察是否能满足以下目标：

1. 支持 Windows 桌面软件自动化，尤其是传统桌面客户端、金融终端、数据查询软件、企业内部系统客户端等。
2. 支持 AI 对桌面界面进行探索，能够根据任务目标识别入口、菜单、输入框、按钮、表格、弹窗等关键 UI 元素。
3. 能将探索、示教或录制得到的流程固化为脚本、工作流或结构化步骤，而不是每次都完全依赖大模型实时决策。
4. 固化后的脚本支持参数化，把某些业务输入值抽象为变量，便于批量传参执行。
5. 脚本能够稳定回放，具备等待、重试、异常检测、日志、截图、结果校验等工程化能力。
6. 尽量减少人工参与，人工主要负责定义任务目标、确认关键步骤、维护少量异常规则，而不是每次手工操作。

## 3. 参数化需求说明

参数化是本次调研的关键要求之一。所谓参数化，是指把录制或探索过程中出现的具体业务输入值抽象成变量，使同一套流程可以复用于不同数据。

例如，用户在某一步输入了：

```text
688697.SH
```

系统在生成脚本时，不应只固化为固定文本 `688697.SH`，而应识别这是一个可替换的业务参数，例如：

```python
stock_code = "{{stock_code}}"
```

或：

```yaml
params:
  stock_code: 688697.SH
steps:
  - input: "{{stock_code}}"
```

这样后续可以通过批量参数传入：

```text
688697.SH
600000.SH
000001.SZ
```

对同一个桌面软件流程进行重复执行，完成不同证券代码、客户编号、产品编号、日期区间、查询条件等数据的获取操作。

调研时需要重点确认：

1. 框架是否支持把录制步骤中的输入值提取为变量。
2. 是否支持参数文件、命令行参数、接口参数、表格参数或数据库参数批量输入。
3. 是否支持循环执行、失败重试、断点续跑和结果汇总。
4. 参数变化后，脚本是否仍然能稳定定位 UI 元素，而不是依赖固定坐标。
5. 参数值导致界面状态变化、弹窗变化、查询结果为空时，是否有可靠的分支处理能力。

## 4. 理想工作流

目标方案最好能够形成如下闭环：

1. 用户输入自然语言任务目标，例如“在某桌面软件中查询一批证券代码的数据并导出结果”。
2. AI Agent 自动观察 Windows 桌面界面，识别当前应用、窗口、菜单和控件。
3. Agent 自主探索或在少量人工示教下完成一次样例流程。
4. 系统记录操作路径、关键 UI 定位方式、输入参数、输出结果和异常情况。
5. 系统生成可审阅、可修改、可版本管理的脚本或工作流。
6. 用户或系统将样例输入值标记为参数，例如证券代码、日期、账号、查询条件等。
7. 脚本读取批量参数，自动完成多次回放执行。
8. 执行过程中自动处理等待、弹窗、错误提示、网络延迟、软件卡顿、查询无结果等情况。
9. 执行结束后输出日志、截图、结果文件、成功失败明细和可追溯报告。

## 5. 调研对象范围

可以纳入调研的对象包括但不限于：

1. AI GUI Agent 框架：具备截图理解、界面探索、自然语言任务执行能力。
2. 桌面自动化框架：如 UI Automation、pywinauto、pyautogui、OCR、图像识别、WinAppDriver 等。
3. RPA 开源框架：能够录制、编排和回放桌面操作流程。
4. 录制回放工具：能够从人工操作中生成脚本、流程或步骤定义。
5. 测试自动化框架：具备稳定定位、断言、日志和报告能力。
6. 能力组合方案：例如“AI Agent 负责探索 + pywinauto 负责执行 + Robot Framework 负责批量回放和报告”。

不要求所有能力都来自同一个项目。如果单一框架无法完整满足要求，可以重点调研组合方案的可行性和工程落地复杂度。

## 6. 重点评价维度

调研每个框架、软件或组合方案时，建议至少从以下维度评估：

| 维度 | 需要回答的问题 |
| --- | --- |
| Windows 桌面支持 | 是否支持真实 Windows 桌面软件？是否支持非浏览器客户端？ |
| UI 定位能力 | 是依赖坐标、图像、OCR、Accessibility/UIA 控件树，还是多种方式组合？ |
| AI 探索能力 | 是否能根据任务目标自主探索界面？是否需要人工先完整示教？ |
| 录制能力 | 是否能记录人工操作？记录的是坐标、控件、截图、事件还是结构化步骤？ |
| 脚本生成能力 | 是否能生成 Python、JS、YAML、Robot、XAML 等可审阅脚本或工作流？ |
| 参数化能力 | 是否能把输入值抽象为变量？是否支持批量参数、循环和数据驱动执行？ |
| 稳定回放能力 | 是否支持等待、重试、断言、异常处理、弹窗处理和恢复机制？ |
| 可维护性 | 脚本是否可读、可调试、可版本管理？UI 变化后维护成本如何？ |
| 私有化部署 | 是否能本地部署？是否依赖云端模型或商业服务？ |
| 开源与社区 | 是否开源？License 是否友好？社区活跃度、Stars、更新频率如何？ |
| 集成成本 | 与现有 Python、数据库、文件系统、调度系统、日志系统集成是否方便？ |
| 生产化能力 | 是否支持任务队列、并发执行、权限隔离、运行报告、失败重跑？ |

## 7. 关键验收标准

建议用一个典型桌面软件任务作为 PoC，用以下标准判断方案是否可行：

1. 能在 Windows 桌面软件中完成一次从打开软件、进入查询页面、输入业务参数、触发查询、读取或导出结果的完整流程。
2. 样例输入值能够被替换为参数，例如证券代码、日期区间或查询条件。
3. 能通过参数列表批量执行至少 10 条不同输入数据。
4. 执行过程中不需要人工逐条干预。
5. 对常见异常有处理策略，例如软件启动慢、控件未加载、查询无结果、弹窗提示、网络延迟、导出文件已存在等。
6. 生成的脚本或工作流可被人工审阅和修改，而不是完全黑盒。
7. 回放成功率、失败原因、执行日志和输出结果可追溯。

## 8. 优先考虑的方案形态

如果没有单一开源项目能完整覆盖全部能力，优先考虑以下组合路线：

1. AI GUI Agent 用于探索和理解界面，例如识别菜单路径、按钮含义、输入框位置和业务流程。
2. Windows UI 自动化框架用于稳定执行，例如 pywinauto、UI Automation、WinAppDriver 或相关封装。
3. OCR 和视觉识别作为补充，用于处理控件树不可见、老旧客户端、自绘界面或图表区域。
4. 脚本/DSL 层用于固化流程，例如 Python、Robot Framework、YAML 或自定义工作流定义。
5. 批量执行层用于参数管理、任务调度、日志记录、失败重试和结果归档。

也就是说，AI 不一定要承担每一次生产执行。更理想的方式是：AI 负责探索、辅助生成和维护脚本；固化脚本负责稳定、可控、可追溯地批量回放。

## 9. 需要避免的误区

调研时需要注意以下风险：

1. 只支持 Web 自动化但不支持 Windows 桌面软件的方案，不适合作为主方案。
2. 只依赖屏幕坐标的录制回放方案稳定性较差，窗口大小、分辨率、缩放比例变化后容易失败。
3. 只依赖大模型实时操作的方案可解释性和稳定性不足，不适合作为高频批量生产任务的唯一执行方式。
4. 只能录屏或保存视频，但不能生成可执行脚本的工具，不满足本次核心目标。
5. 参数化能力弱、无法批量传入不同业务数据的方案，不适合数据采集和批量查询场景。
6. 完全闭源或强依赖商业云服务的方案，需要单独评估成本、合规、私有化和可控性。

## 10. 建议输出格式

每个被调研对象建议形成一份独立文档，包含：

1. 项目名称、官网、GitHub 地址、License、Stars、最近更新时间。
2. 主要能力概述。
3. 是否支持 Windows 桌面软件。
4. 是否支持 AI 探索或语义理解。
5. 是否支持录制、示教或自动生成流程。
6. 是否支持生成可审阅脚本或工作流。
7. 是否支持参数化和批量执行。
8. 是否支持稳定回放、异常处理、日志和报告。
9. 与本需求的匹配度评分。
10. 适合作为主方案、辅助方案、执行层、探索层还是不推荐。
11. 主要优点、短板和落地风险。
12. 推荐 PoC 验证方式。

## 11. 总体判断口径

本次调研的最终结论不应只看项目热度或单点能力，而应重点判断它能否支撑以下闭环：

```text
AI 理解任务 -> 探索 Windows 桌面软件 -> 形成可审阅流程 -> 参数化业务输入 -> 批量稳定回放 -> 输出可追溯结果
```

越接近这个闭环的方案，优先级越高。若单个项目无法完成闭环，则应明确推荐能力组合，并说明每一层分别由什么工具承担。
