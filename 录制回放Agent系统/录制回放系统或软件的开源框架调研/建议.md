# 录制回放与 CS 客户端自动化建议

剔除“只支持 Web/浏览器”的工具后，剩下可以考虑的主要是下面这些。这里的“支持 CS 客户端”按能操作 Windows/macOS/Linux 本地桌面软件、传统客户端、Office、ERP 客户端、Java/SAP/WinForms/WPF/Win32 等软件界面来理解。

## 一、优先级结论

### 最适合原始需求的前五个

| 推荐度 | 工具 | 适合程度 |
| ---: | --- | --- |
| 1 | UI.Vision RPA | 最接近“录制、回放、参数化、桌面+网页兼顾” |
| 2 | OpenRPA | 更像传统 RPA Studio，适合 Windows 客户端流程 |
| 3 | AutoHotkey + Pulover’s Macro Creator | Windows 本地软件宏录制和脚本化最实用 |
| 4 | TagUI | 参数化、批量运行强，但桌面端更多依赖视觉自动化 |
| 5 | Robot Framework + RPA Framework | 长期维护、参数化、日志报告强，但录制能力弱 |

### 更偏 AI 自动操作电脑的工具

| 推荐度 | 工具 | 适合程度 |
| ---: | --- | --- |
| 1 | OpenAdapt | 最像“看你操作一遍，然后学习自动化” |
| 2 | UI-TARS Desktop | AI 直接看屏幕、控制电脑，偏前沿 |
| 3 | Microsoft UFO | 专门面向 Windows 应用的 GUI Agent |
| 4 | Agent S | 开源 GUI Agent 框架，研究/二次开发价值高 |
| 5 | CUA / trycua | 适合搭建 computer-use agent 基础设施 |

---

## 二、保留下来的开源软件清单

### 1. UI.Vision RPA

UI.Vision 是开源 RPA 工具，不只是浏览器自动化，也支持 Windows、macOS、Linux 上的桌面应用自动化；它还支持 CSV 读写，适合数据驱动的参数化执行。

**优点**

- 有录制、回放能力，上手快。
- 同时支持网页和桌面软件。
- 支持图像识别、OCR、鼠标键盘模拟。
- 支持 CSV 参数化，适合同一流程批量替换单位名称、账号、文件路径、表单字段。
- 非程序员也能较快做出第一个可运行脚本。

**缺点**

- 桌面自动化主要依赖视觉识别和坐标/图像匹配，窗口缩放、分辨率、主题变化可能影响稳定性。
- AI 不是它的核心能力，更多是 RPA + 视觉自动化。
- 复杂业务判断、异常弹窗处理仍需要脚本设计。
- 对大型企业级流程编排，不如专业 RPA 平台严整。

**适合场景**

- 先快速做原型。
- 需要录一次、改参数、批量回放。
- 同时涉及网页和桌面客户端。

**推荐度：★★★★★**

---

### 2. OpenRPA

OpenRPA 是开源 RPA 软件，支持录制器、设计器、选择器等传统 RPA 能力；官方介绍中提到可自动化 Windows、SAP、Java、浏览器、主机、图像和 Microsoft Office，并支持在流程中嵌入 C#、VB、Python、PowerShell、AutoHotkey 等脚本。

**优点**

- 更接近 UiPath 这类传统 RPA Studio 的思路。
- 适合 Windows 客户端、Office、SAP、Java 客户端等场景。
- 有流程设计器、录制器、变量、工作流。
- 可与 OpenFlow / OpenCore 做调度和机器人管理。
- 比单纯宏录制更适合复杂流程。

**缺点**

- 安装、配置、调试门槛比 UI.Vision 高。
- 项目生态和商业工具相比小一些。
- AI 能力不是内置核心，需要额外接 LLM。
- 对极复杂界面仍可能需要图像识别或脚本补充。

**适合场景**

- Windows 内部业务系统。
- 老 ERP、SAP、Java 客户端。
- 需要流程化、低代码、变量和调度。

**推荐度：★★★★★**

---

### 3. AutoHotkey + Pulover’s Macro Creator

AutoHotkey 是 Windows 上免费开源的自动化脚本语言，可写热键、鼠标键盘、窗口控制等脚本；Pulover’s Macro Creator 是基于 AutoHotkey 的免费宏录制和脚本生成工具，有内置录制器，可以导出为 AutoHotkey 脚本。

**优点**

- Windows 桌面软件自动化非常实用。
- Pulover 可以录制操作，AutoHotkey 可以把录制结果脚本化。
- 参数化灵活，可通过变量、配置文件、命令行参数、Excel/CSV 读取实现。
- 很适合让 AI 帮你改脚本。
- 轻量，不需要复杂平台。

**缺点**

- 主要面向 Windows。
- 录制出来的脚本容易偏坐标化，需要后期优化。
- 对复杂控件识别不如 pywinauto / FlaUI / OpenRPA。
- 没有天然的流程管理、日志审计和机器人调度。

**适合场景**

- Windows 客户端重复点击、复制粘贴、快捷键操作。
- 本地文件夹、Excel、客户端系统之间的简单流程。
- 个人办公自动化。

**推荐度：★★★★★**

---

### 4. TagUI

TagUI 是 AI Singapore 维护的开源 RPA 工具，定位是自动化重复性或时间敏感任务，目标包括桌面和网站上的用户界面操作；它默认运行在用户本机，不是云端 SaaS。

**优点**

- 脚本语法相对接近自然语言。
- 参数化和批量执行能力强，特别适合 CSV 数据驱动。
- 可在 Windows、macOS、Linux 上运行。
- 可结合视觉自动化处理桌面软件。
- 比纯录制宏更利于维护。

**缺点**

- 桌面客户端自动化能力不如 OpenRPA 那样完整。
- 图形化设计器能力较弱。
- 复杂桌面控件识别要依赖视觉识别、Sikuli 或额外脚本。
- 项目活跃度和生态不如 Playwright、Robot Framework 这类通用框架。

**适合场景**

- 批量填报。
- 参数化任务。
- 技术人员维护的轻量 RPA。
- 桌面 + 网页混合流程，但桌面端不要太复杂。

**推荐度：★★★★☆**

---

### 5. Robot Framework + RPA Framework

Robot Framework 是开源自动化框架，可用于测试自动化和 RPA；RPA Framework 是面向 Robot Framework 和 Python 的开源 RPA 库集合，其中包含 Windows 桌面自动化库。

**优点**

- 参数化、变量、数据驱动、日志、报告能力非常强。
- 适合长期维护、多人协作、版本管理。
- 可连接 Python、Excel、PDF、邮件、文件系统、Windows 自动化等。
- 脚本结构清晰，适合形成单位内部标准化自动化资产。
- 可以把 AI 生成的流程固化成正式脚本。

**缺点**

- 本身不是强录制器。
- 初学门槛比 UI.Vision / Pulover 高。
- 桌面客户端自动化需要额外库和调试。
- 对普通办公人员不如低代码 RPA 直观。

**适合场景**

- 稳定运行的长期任务。
- 批量自动化。
- 需要日志、报告、审计。
- 技术人员维护的正式自动化工程。

**推荐度：★★★★☆**

---

### 6. SikuliX / Oculix

SikuliX 的核心是“自动化你在屏幕上看到的东西”，通过图像识别来定位并操作界面；其文档显示它支持用 IDE 创建、维护视觉工作流，并支持 Python、Ruby、JavaScript 等脚本。

**优点**

- 非常适合没有接口、没有标准控件、无法用选择器识别的老软件。
- 对远程桌面、虚拟机、老式业务系统、图形化客户端很有用。
- 可以通过截图匹配按钮、菜单、图标。
- 支持脚本化和参数化。

**缺点**

- 对分辨率、缩放比例、主题颜色、窗口位置敏感。
- 录制后通常需要人工整理图片和脚本。
- 不是真正 AI，只是计算机视觉匹配。
- 大规模稳定运行需要大量异常处理。

**适合场景**

- 传统 CS 客户端无法被控件识别。
- 老软件、行业专用软件、远程桌面。
- 作为 OpenRPA、TagUI、Robot Framework 的视觉补充。

**推荐度：★★★★☆**

---

### 7. PyAutoGUI

PyAutoGUI 是跨平台 Python GUI 自动化库，可以控制鼠标和键盘，也可以截图；官方文档说明它可在 Windows、macOS、Linux 上自动化其他应用。

**优点**

- Python 生态强，适合和 AI、OCR、Excel、数据库、文件处理结合。
- 参数化非常灵活。
- 适合由 AI 生成脚本、由人稍作修正。
- 跨平台。
- 轻量，容易嵌入自己的自动化程序。

**缺点**

- 没有成熟录制器。
- 主要依赖坐标、图片、鼠标键盘模拟。
- 稳定性取决于脚本设计水平。
- 不适合完全非技术人员直接使用。

**适合场景**

- 技术人员用 Python 写桌面自动化。
- AI 辅助生成脚本。
- 需要同时处理 Excel、文件、OCR、客户端点击。

**推荐度：★★★☆☆ 到 ★★★★☆**

---

### 8. pywinauto

pywinauto 是 Windows GUI 自动化 Python 库，支持 Win32 API 和 Microsoft UI Automation 后端，可以操作 Windows 对话框和控件，也能发送鼠标键盘动作。

**优点**

- 很适合 Windows 原生客户端。
- 比纯坐标点击稳定，因为可以识别控件。
- Python 生态好，适合参数化和 AI 辅助生成脚本。
- 对 Win32、WPF、WinForms 等传统 Windows 应用较有价值。

**缺点**

- 只适合 Windows 桌面自动化。
- 没有面向普通用户的录制器。
- 对自绘界面、游戏式界面、远程桌面效果有限。
- 需要一定编程能力。

**适合场景**

- Windows CS 客户端。
- 内部管理软件。
- 需要比图像识别更稳的控件级自动化。

**推荐度：★★★★☆**

---

### 9. FlaUI

FlaUI 是 .NET 的 Windows UI Automation 库，支持 Win32、WinForms、WPF、Store Apps 等 Windows 应用自动化测试。

**优点**

- 对 Windows 桌面应用控件识别较强。
- 适合 C#/.NET 技术栈。
- 比坐标点击更稳定。
- 可做自动化测试，也可做业务流程自动化。
- 现在还有 FlaUI-MCP 这类把 FlaUI 接入 AI Agent 的探索项目。

**缺点**

- 没有普通用户友好的录制回放界面。
- 需要 C#/.NET 编程能力。
- 对自绘控件、非标准界面支持有限。
- 偏测试框架，不是完整 RPA 平台。

**适合场景**

- .NET / WPF / WinForms 客户端。
- Windows 软件测试自动化。
- 想把 AI Agent 接到 Windows 控件树上。

**推荐度：★★★★☆**

---

### 10. WinAppDriver

WinAppDriver 是微软开源的 Windows 应用自动化服务，支持 UWP、WinForms、WPF、Classic Win32 等 Windows 应用，并采用类似 Selenium 的方式进行 UI 自动化。

**优点**

- 面向 Windows 应用，技术路线清晰。
- Selenium/Appium 风格，测试人员容易理解。
- 支持常见 Windows 客户端技术栈。
- 适合自动化测试和部分 RPA 流程。

**缺点**

- 项目维护活跃度需要谨慎评估。
- 配置和调试比 pywinauto / FlaUI 更繁琐。
- 不适合非技术人员。
- 对复杂业务 RPA、参数管理、流程编排支持较弱。

**适合场景**

- Windows 客户端测试自动化。
- 已有 Selenium/Appium 技术基础的团队。
- 稳定控件可识别的 CS 软件。

**推荐度：★★★☆☆**

---

### 11. Taskt

Taskt 是 .NET/C# 编写的免费开源 RPA 客户端，目标是让用户无需写代码也能设计流程自动化。

**优点**

- 免费开源。
- 有低代码/无代码思路。
- 适合 Windows 上简单 RPA。
- 入门比纯代码工具友好。

**缺点**

- 生态和活跃度相对有限。
- AI 能力不强。
- 复杂流程、参数化、异常恢复能力不如 OpenRPA / Robot Framework。
- 不建议作为关键生产流程的首选。

**适合场景**

- 学习 RPA。
- 简单 Windows 桌面流程。
- 小范围个人自动化。

**推荐度：★★★☆☆**

---

### 12. LDTP

LDTP 是较老的开源桌面 GUI 测试工具，基于辅助技术进行 GUI 自动化，可运行在 Linux、Windows、macOS、Solaris、FreeBSD 等环境。

**优点**

- 面向桌面 GUI 自动化。
- 支持多平台。
- 对 Linux 桌面应用历史上较有价值。
- 可用于一些自动化测试场景。

**缺点**

- 项目年代较久，活跃度弱。
- 文档和生态不如现代工具。
- 不适合作为新项目首选。
- AI、录制、参数化体验都不突出。

**适合场景**

- Linux 桌面自动化测试的历史项目维护。
- 特定兼容性需求。

**推荐度：★★☆☆☆**

---

## 三、AI 驱动类：能操作电脑，但录制/参数化通常不成熟

### 13. OpenAdapt

OpenAdapt 是开源的 AI-first 流程自动化平台，定位是连接大型多模态模型和传统桌面/Web GUI，可以录制 GUI 演示、训练模型、评估 Agent；官网也明确说它可以记录人工演示、训练模型并部署能操作任意软件的 Agent。

**优点**

- 最接近“AI 看我操作一遍，然后学习自动执行”。
- 支持桌面 GUI 和 Web GUI。
- 有录制人类演示的思路。
- 适合探索“AI 驱动 RPA”的未来路线。

**缺点**

- 官方也提示属于 alpha 软件，稳定性和成熟度要谨慎看待。
- 参数化脚本回放不如传统 RPA 清晰。
- 需要多模态模型支持。
- 生产落地风险较高。

**适合场景**

- 研究型、探索型。
- 想尝试 AI 通过演示学习桌面软件操作。
- 非关键任务试点。

**推荐度：★★★★☆，但不建议单独用于关键生产流程。**

---

### 14. UI-TARS Desktop

UI-TARS Desktop 是字节开源的桌面 GUI Agent，提供本地和远程电脑操作，以及浏览器操作，目标是通过多模态大模型实现更接近人类的任务完成流程。

**优点**

- 真正偏“AI 操作电脑”。
- 可看屏幕、理解界面、执行鼠标键盘操作。
- 更适合处理动态界面和非固定流程。
- 适合本地模型或多模态模型探索。

**缺点**

- 不是传统 RPA 录制器。
- 参数化、稳定回放、日志审计需要额外工程化。
- 对模型能力、显卡/API、权限要求较高。
- 长流程准确率仍需实际验证。

**适合场景**

- AI 控制电脑实验。
- 低风险桌面任务。
- 与 OpenRPA / AHK / PyAutoGUI 组合使用。

**推荐度：★★★☆☆ 到 ★★★★☆**

---

### 15. Microsoft UFO

UFO 是微软开源的 Windows UI-Focused Agent，论文介绍其面向 Windows OS 应用操作，可观察和分析 Windows 应用 GUI 与控件信息，并跨应用执行任务。

**优点**

- 专门针对 Windows 桌面应用。
- 强调 GUI 观察、控件信息和跨应用操作。
- 比单纯浏览器 Agent 更适合 CS 客户端。
- 适合研究 Windows AI 自动化。

**缺点**

- 不是传统录制回放工具。
- 参数化脚本体系不成熟。
- 更偏 Agent 研究框架。
- 实际落地需要结合 RPA/脚本工具。

**适合场景**

- Windows AI Agent 研究。
- 希望 AI 根据自然语言操作多个 Windows 软件。
- 与 pywinauto/FlaUI/OpenRPA 结合做增强。

**推荐度：★★★☆☆**

---

### 16. Agent S

Agent S 是开源框架，目标是通过 Agent-Computer Interface 让智能体像人一样与电脑 GUI 交互，执行复杂任务。

**优点**

- 代表开源 GUI Agent 的前沿方向。
- 支持复杂、多步骤、动态界面任务研究。
- 有经验记忆、层级规划等 Agent 思路。
- 对构建自己的 AI 桌面自动化框架有参考价值。

**缺点**

- 不是开箱即用的办公 RPA。
- 没有成熟“录制脚本—参数化回放”体验。
- 部署和调试门槛较高。
- 适合技术团队，不适合普通用户直接使用。

**适合场景**

- AI Agent 研发。
- 桌面自动化研究。
- 作为底层框架二次开发。

**推荐度：★★★☆☆**

---

### 17. CUA / trycua

trycua/cua 是开源 Computer-Use Agent 基础设施，提供沙箱、SDK、基准测试，可训练和评估控制完整桌面的 AI Agent，覆盖 macOS、Linux、Windows。其 Cua Driver 也定位为原生桌面应用的后台 computer-use driver，可让 Agent 点击、输入、滚动、检查可访问性树并捕获窗口状态。

**优点**

- 很适合搭建安全、隔离、可评估的 AI 电脑操作环境。
- 支持完整桌面，不只浏览器。
- 适合技术团队做平台化。
- 对 Agent 测试、训练、评估有价值。

**缺点**

- 不是普通用户录制器。
- 参数化回放要自己设计。
- 更偏基础设施，不是完整 RPA 产品。
- 落地复杂度较高。

**适合场景**

- 自建 AI 桌面 Agent 平台。
- 沙箱化执行电脑任务。
- 研发团队评测不同模型操作电脑的能力。

**推荐度：★★★☆☆ 到 ★★★★☆**

---

### 18. Open Computer Use / Coasty

Open Computer Use 是开源平台，目标是让 AI Agent 具备真实电脑控制能力，可浏览网页、运行命令、点击 UI、编排多步工作流，并支持 Windows、macOS、Linux 原生自动化。

**优点**

- 方向非常接近“AI 操作电脑”。
- 不局限网页，可操作桌面和终端。
- 适合自然语言驱动多步骤任务。
- 适合技术团队试验 computer-use agent。

**缺点**

- 传统录制、脚本参数化、稳定回放能力不是核心。
- 生产级安全、权限、日志、回滚需要额外设计。
- 对模型能力依赖较强。
- 生态成熟度还要观察。

**适合场景**

- AI 电脑操作原型。
- 桌面 + 浏览器 + 命令行混合任务。
- 非关键流程探索。

**推荐度：★★★☆☆**

---

### 19. Windows-MCP / MCP Windows Automation / FlaUI-MCP

Windows-MCP 这类项目通过 MCP 让 AI 控制 Windows，通常包含鼠标、键盘、窗口状态、UI 自动化等工具；有的项目强调不依赖传统计算机视觉，而是结合 Windows UI 状态。FlaUI-MCP 则把 FlaUI 封装为 MCP Server，让 AI Agent 通过可访问性 API 自动化 Windows 桌面应用。

**优点**

- 很适合接入 AI 助手。
- 面向 Windows 客户端更直接。
- 可以结合 UI Automation，比纯截图点击更稳。
- 适合做“AI + 本地 Windows 软件”的工具层。

**缺点**

- 不是完整 RPA 产品。
- 通常没有成熟录制器。
- 参数化、日志、审批、回放要自己封装。
- 权限安全要特别小心。

**适合场景**

- 让 AI 调用 Windows 软件。
- 结合 Claude/Cursor/本地 LLM 做桌面操作。
- 技术人员自建自动化助手。

**推荐度：★★★☆☆**

---

### 20. Open Interpreter

Open Interpreter 是开源的自然语言电脑操作工具，主要让 LLM 在本地运行代码、操作文件、执行脚本；其官网也将其描述为可以读、改、创建本机文档的桌面 Agent。

**优点**

- 很适合让 AI 生成并执行 Python、Shell、AHK、PyAutoGUI 脚本。
- 文件、Excel、PDF、数据处理能力强。
- 适合做自动化脚本助手。
- 可作为“AI 编程 + 本地执行”的中间层。

**缺点**

- 不是真正专门的 GUI 录制回放工具。
- 直接操作复杂 CS 客户端不是强项。
- 需要严格控制权限。
- 参数化和回放要靠自己写脚本。

**适合场景**

- 让 AI 帮你写自动化脚本。
- 本地文件、数据、命令行、部分桌面任务。
- 与 PyAutoGUI / AutoHotkey / Robot Framework 配合。

**推荐度：★★★☆☆**

## 六、最终建议

如果目标是真正可用、可回放、可参数化、能操作 CS 客户端，建议不要一开始就追求“完全 AI 自主操作”。更稳的架构是：

| 能力层 | 建议工具 |
| --- | --- |
| 录制/固化流程 | UI.Vision / OpenRPA / Pulover / TagUI |
| 参数化数据 | Excel / CSV / JSON |
| 桌面控件增强 | pywinauto / FlaUI / SikuliX |
| AI 辅助 | OpenAdapt / UI-TARS / UFO / Windows-MCP |

### 最实用落地组合

**方案一：UI.Vision RPA**

适合：最快试用、录制回放、桌面+网页混合。

**方案二：OpenRPA + SikuliX**

适合：Windows 客户端、老系统、复杂流程。

**方案三：AutoHotkey + Pulover + pywinauto**

适合：Windows 办公软件、内部客户端、轻量高效。

**方案四：Robot Framework + RPA Framework + pywinauto/FlaUI**

适合：长期维护、批量执行、需要日志和审计。

**方案五：OpenAdapt / UI-TARS Desktop + 传统 RPA**

适合：探索 AI 自主操作电脑，但要让传统 RPA 负责稳定执行。

一句话：要稳定，就选 UI.Vision / OpenRPA / AutoHotkey；要工程化，就选 Robot Framework + pywinauto/FlaUI；要 AI 探索，就选 OpenAdapt / UI-TARS / UFO，但最好不要单独依赖它们做关键任务。
