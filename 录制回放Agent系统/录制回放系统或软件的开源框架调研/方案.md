你现在最该做的，不是继续证明 **OculiX 本身没问题**，而是换一种说法：

> **OculiX 不应该作为“最终产品形态”，而应该作为“稳定回放底座/兜底执行器”。领导要的是上层的“自动录制、自动参数化、自动生成脚本”。**

你们其实在争论两个不同层级的东西。

领导要的是：

> 人点一遍流程 → 系统自动理解步骤 → 自动识别哪些输入值可参数化 → 生成可批量执行的稳定脚本。

而 OculiX 现在提供的是：

> 人工准备图片定位 → 手写/调整脚本 → 稳定回放。

所以领导否掉 OculiX，不一定是说它技术不行，而是觉得它**不像产品，不像平台，人工介入太重**。

---

## 我建议你走“混合架构”，不要押单一框架

最适合你现在场景的方向是：

> **Recorder 负责自动采集；AI 负责理解和参数化；OculiX/FlaUI/UIA 负责稳定执行。**

也就是把系统拆成三层：

```text
第一层：录制层
用户正常点击、输入、回车、选择下拉框
系统自动记录：
- 鼠标/键盘事件
- 当前窗口标题
- 控件 UIA 信息
- 点击位置附近截图
- 输入文本
- 时间顺序

第二层：脚本生成层
AI/规则分析录制日志：
- 哪些是固定动作
- 哪些是参数，比如 688697、20241230、outputFileName
- 哪些步骤可以循环
- 哪些定位方式最稳定

第三层：执行层
优先用：
- UIA/FlaUI 定位控件
- 找不到控件时用 OculiX 图片识别兜底
- 再不行才用窗口相对坐标
```

这才是领导真正想要的东西。

---

## 你不要把 OculiX 扔掉，而是降低它的角色

OculiX 的问题不是“回放不行”，而是“录制生成脚本不够自动”。

所以你可以把它变成：

> **兜底定位引擎**。

比如用户录制时点了一个按钮，你的 Recorder 自动做这些事：

1. 记录点击坐标。
2. 读取当前窗口 UIA 控件树。
3. 自动截取点击点周围的小图。
4. 保存成一个 step。
5. 生成执行脚本时，优先用 UIA 定位；UIA 定位失败，再用 OculiX 图片定位。

这样 OculiX 最大的缺点“人工取截图”，就被你们自己的录制器补掉了。

你现在的问题不是缺执行器，而是缺一个 **“录制到脚本的编译层”**。

## 现成框架为什么很难直接满足你领导的要求

现在几类方案大概是这样：

**Windows-Use** 这类方案偏 AI Agent，它通过 Windows UI Automation API 读取界面，再让 LLM 决定点击、输入、滚动等操作，不依赖视觉模型。它适合“AI 自己操作电脑”，但不等于天然能生成企业可审计、可参数化、可稳定回放的脚本。([GitHub][1])

**OpenAdapt** 的定位更接近“录制 GUI demonstration，然后训练/评估 agent”，它确实有录制人类演示的方向，但它更偏 AI-first automation / agent 学习，不是你要的那种 Wind/同花顺里录一遍后直接生成稳定参数化 RPA 脚本。([GitHub][2])

**UI-TARS Desktop** 是多模态 GUI Agent，主要靠截图理解界面并执行鼠标键盘操作，适合探索式自动化，但它也不是传统意义上的“确定性脚本生成器”。([GitHub][3])

**Power Automate Desktop** 反而最像领导脑子里的产品形态：它有桌面录制器，也支持变量、桌面流等能力。微软文档也明确有 Recorder 和变量体系。([微软学习][4]) 但问题是它不是你要的开源方案，而且后续企业集成、内网部署、授权、可控性都会受限制。

所以我的判断是：**开源世界里，暂时没有一个东西能完整满足“零人工截图 + 自动参数化 + 企业稳定回放 + Windows 金融客户端适配”这几个条件。**

你要做的是组合，而不是寻找一个完美框架。

## 你可以给领导一个更容易接受的方案

不要说：

> 我们继续用 OculiX。

这样领导会觉得你在坚持旧方案。

你应该说：

> 单独使用 OculiX 确实不够产品化，因为它依赖人工截图和脚本调整。我的新方案是做一个上层录制器，录制时自动采集 UIA 信息、点击区域截图和输入值，再由脚本生成器自动生成参数化流程。OculiX 只作为执行层兜底能力，保证金融客户端在 UIA 不稳定时仍然可以稳定回放。

这个说法会完全不一样。

## 推荐你做的最小可行版本

第一版不要追求“AI 全自动理解所有流程”。你可以做一个比较务实的版本：

### 1. 录制时自动采集

用户点击、输入时，系统记录成 JSON：

```json
[
  {
    "type": "click",
    "windowTitle": "Wind金融终端..Alice",
    "x": 1520,
    "y": 930,
    "uia": {
      "name": "键盘精灵",
      "controlType": "Button",
      "automationId": ""
    },
    "image": "step_001_click_target.png"
  },
  {
    "type": "input",
    "value": "EDE"
  },
  {
    "type": "input",
    "value": "688697"
  },
  {
    "type": "key",
    "value": "ENTER"
  }
]
```

### 2. 参数识别

系统发现这些输入值：

```text
EDE
20251230
688697
```

然后让用户确认：

```text
688697 是否抽成参数 stockCode？
20251230 是否抽成参数 reportDate？
EDE 是否保持固定？
```

这里不需要 AI 百分百自动。第一版只要做到“自动推荐 + 人点确认”，领导一般就能接受，因为这已经不是人工写代码了。

### 3. 生成中间 DSL

不要一上来直接生成 Python/OculiX 脚本。先生成你们自己的流程描述：

```json
{
  "name": "wind_dividend_indicator_query",
  "params": ["stockCode", "reportDate", "outputFileName"],
  "steps": [
    {
      "action": "focusWindow",
      "titleContains": "Wind金融终端"
    },
    {
      "action": "click",
      "target": {
        "uiaName": "键盘精灵",
        "image": "step_001_click_target.png"
      }
    },
    {
      "action": "type",
      "text": "EDE"
    },
    {
      "action": "press",
      "key": "ENTER"
    },
    {
      "action": "type",
      "text": "${stockCode}"
    },
    {
      "action": "press",
      "key": "ENTER"
    }
  ]
}
```

这个 DSL 才是你们平台的核心资产。

以后可以编译成：

```text
DSL → OculiX 脚本
DSL → FlaUI 脚本
DSL → AutoHotkey 脚本
DSL → Python pywinauto 脚本
```

这样你不会被某个框架绑死。

## 对你当前项目，我最推荐的技术路线

你可以按这个优先级来做：

```text
录制层：
Python pynput / pywin32 / UIAutomation / pywinauto / FlaUI

控件识别：
优先 UIA 树
失败时自动截图
再失败时窗口相对坐标

执行层：
优先 FlaUI / pywinauto
图像兜底 OculiX
必要时 AutoHotkey 兜底键鼠

脚本格式：
自定义 JSON/YAML DSL

参数化：
规则 + AI 辅助
例如股票代码、日期、文件名、指标名自动识别为候选参数

平台接口：
FastAPI 接收 JSON
执行 DSL 或编译后的脚本
```

OculiX 在这里不是主角，但非常有价值。尤其是 Wind/同花顺这种金融客户端，很多控件不一定有稳定的 UIA 信息，图片识别兜底反而很重要。

## 你现在可以怎么汇报

你可以这样讲：

> 之前调研的 OculiX 在稳定回放方面是有价值的，但它单独使用时确实存在人工截图和脚本调整成本，不适合作为完整产品方案。
>
> 我建议调整为混合方案：前端做录制器，用户只需要正常操作一遍流程；录制器自动采集鼠标键盘、窗口、UIA 控件信息和点击区域截图；中间层生成可参数化的 DSL 流程；执行层优先使用 UIA/FlaUI 回放，失败时使用 OculiX 图片识别兜底。
>
> 这样既满足领导要求的“少人工参与、可参数化生成脚本”，又保留 OculiX 在金融客户端复杂界面中的稳定识别能力。

---

## 最关键的一句话

你不要再把目标定义成：

> 找一个录制回放框架。

要改成：

> 做一个“录制行为 → 生成参数化 DSL → 多执行器稳定回放”的平台。

这个方向才对。

OculiX 不是答案，但它可以是答案的一部分。你真正要补的是：**自动录制、自动截图、自动参数识别、DSL 生成器**。
