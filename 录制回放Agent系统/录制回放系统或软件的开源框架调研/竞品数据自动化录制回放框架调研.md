# 录制回放系统或软件的开源框架调研清单

## 排序依据

本清单按当前任务目标排序：优先寻找**支持桌面软件、能够 AI 驱动探索、并尽量生成可保存/可回放脚本或工作流**的框架。

排序权重参考：

1. 是否支持桌面软件，尤其是 Windows 桌面应用。
2. 是否具备 AI 语义理解和自动探索能力。
3. 是否能把探索或录制结果保存为可回放资产，最好是 JS、YAML、Python、Robot 或 XAML 等可审阅脚本/工作流。
4. GitHub Stars、社区热度、近期版本/迭代活跃度。
5. 是否开源、是否便于私有化部署和二次开发。

综合匹配度采用 5 星制：`★★★★★` 表示最接近目标，`★★★★☆` 表示值得重点验证，`★★★☆☆` 表示可作为辅助层或备选，`★★☆☆☆` 表示仅适合特定场景。半星用 `½` 表示。

## 组合判断

现有 `pkg` 里的框架单独看都有短板：传统 RPA 稳定但不会自己探索，GUI Agent 会探索但不稳定沉淀脚本，视觉工具能兜底但维护成本高。因此更合理的路线是分层组合：

1. **探索层**：Midscene、Microsoft UFO、UI-TARS Desktop、Agent S、OpenAdapt、CUA/trycua。
2. **轨迹与技能层**：参考 CUA-Skill，把成功运行轨迹抽象成参数化技能图，而不是直接保存坐标序列。
3. **脚本生成层**：由 LangGraph 编排 LLM 生成 DSL、Python、Robot、Playwright、FlaUI、OpenRPA XAML 或 AHK。
4. **稳定执行层**：Web 优先 Playwright；Windows 控件优先 pywinauto/FlaUI/WinAppDriver；流程型 RPA 可选 OpenRPA/Robot Framework；控件不可见时用 OCR/SikuliX/pyautogui 兜底。
5. **验证发布层**：用同一参数集至少回放 3 次，生成日志、截图、结果断言和失败原因，审核通过后进入脚本库。

结论：**不要把 AI Agent 直接作为生产回放执行器，而要把它作为“探索和脚本生成器”。生产阶段应尽量运行确定性脚本。**
